# Event Engine Architecture

## 1. Overview

The event engine is the central nervous system of JWare Solutions. It replaces a linear pipeline (intake, develop, deliver) with a reactive, event-driven loop where company activity emerges from events rather than following a predetermined script. Events trigger responses. Responses generate new events. Multiple workstreams run simultaneously. The result feels like a real company -- people reacting to what just happened, not executing step 14 of a 40-step checklist.

### How It Works

1. Events accumulate in a persistent queue at `.jware/events.json`.
2. Each event has a type, a payload, a source, and a timestamp.
3. When the user invokes `/jware-auto`, the engine processes unprocessed events in priority-then-timestamp order.
4. Processing an event may dispatch personality agents, write to issuetracker, update state files, and generate new events.
5. Generated events are appended to the queue and processed in the same cycle (with recursion limits).
6. The engine updates `.jware/state.json` with the current project phase after each processing cycle.

### Design Principles

- **Reactive, not scripted.** The company responds to what happened, not what was planned to happen next.
- **Observable.** Every state change is traceable to an event. Every event is traceable to a source.
- **Concurrent.** Multiple projects can have events in flight simultaneously. Events from different projects do not block each other.
- **Deterministic.** Given the same event queue and state, the engine produces the same outputs. No hidden randomness in process rules.
- **Recoverable.** A crashed session leaves the queue intact. Unprocessed events are still unprocessed. The engine can resume cleanly.

## 2. Event Schema

Every event in the queue conforms to this structure:

```json
{
  "id": "evt_a7f3b2c1-4e5d-4a8b-9c1e-2f3a4b5c6d7e",
  "type": "review:rejected",
  "source": "margaret-chen",
  "target": null,
  "projectId": "proj_001",
  "payload": {},
  "priority": "MEDIUM",
  "timestamp": "2026-03-24T14:32:00.000Z",
  "processed": false,
  "processedAt": null,
  "processedBy": null,
  "generatedEvents": [],
  "parentEventId": null,
  "depth": 0
}
```

### Field Definitions

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string | yes | UUID prefixed with `evt_`. Generated at creation time. |
| `type` | string | yes | Event type from the type reference (section 3). Format: `domain:action`. |
| `source` | string | yes | Personality slug (e.g., `margaret-chen`) or `system` for engine-generated events, or `customer` for user-initiated events. |
| `target` | string or null | no | Personality slug of the intended recipient, if applicable. Null for broadcast events. |
| `projectId` | string | yes | Project identifier. Format: `proj_NNN`. |
| `payload` | object | yes | Event-type-specific data. Schema varies by type (see section 3). |
| `priority` | string | yes | One of: `CRITICAL`, `HIGH`, `MEDIUM`, `NORMAL`, `LOW`. See section 6. |
| `timestamp` | string | yes | ISO 8601 timestamp. Set at event creation. |
| `processed` | boolean | yes | `false` when created, `true` after the engine has executed the process rule. |
| `processedAt` | string or null | yes | ISO 8601 timestamp of when processing completed. Null until processed. |
| `processedBy` | string or null | no | The personality or system process that handled the event. |
| `generatedEvents` | array of strings | yes | List of `id` values for events this event's processing created. Empty array until processed. |
| `parentEventId` | string or null | yes | The `id` of the event that generated this one. Null for root events (user-initiated or external). |
| `depth` | integer | yes | Recursion depth. Root events are `0`. Events generated by processing a depth-N event are depth `N+1`. Maximum depth: `10`. |

## 3. Complete Event Type Reference

### 3.1 Project Lifecycle Events

---

#### `project:intake`

A new project has been submitted by the customer.

**Trigger:** User describes a new project or the system receives a project request.

**Payload:**
```json
{
  "title": "string — project name",
  "description": "string — what the customer wants built",
  "clientName": "string — customer/client identifier",
  "urgency": "string — low | normal | high | critical",
  "constraints": ["string — any constraints the customer mentioned"],
  "budget": "string or null — budget indication if provided",
  "timeline": "string or null — desired timeline if provided"
}
```

**Process Rule:**
1. Create project directory at `.jware/projects/{projectId}/`.
2. Write `project.json` with initial project metadata (title, description, phase: `intake`, timestamps).
3. Create issuetracker project using the project title.
4. Assign Hannah Reeves (PM) as primary project manager. If Hannah is at capacity (3+ active projects), assign Jordan Pace.
5. Dispatch Daniel Kwon (Solutions Architect) to perform initial technical assessment of the description.
6. Dispatch Ben Hartley (TPM) to create initial project timeline skeleton.
7. Log intake record to `.jware/projects/{projectId}/intake.json`.
8. Generate event: `project:scoping` with Daniel Kwon as source.

**Generated Events:** `project:scoping`

**Visibility Output:**

| Level 1 | Level 2 | Level 3 |
|---------|---------|---------|
| "New project received: {title}" | "New project '{title}' received. Daniel Kwon is running technical discovery. Hannah Reeves assigned as PM." | Full intake details including Daniel's initial technical assessment notes, Hannah's client context brief, and Ben's preliminary timeline skeleton. |

#### `project:scoping`

The scoping phase has begun. The Solutions Architect is evaluating technical feasibility, effort, and architecture.

**Trigger:** Generated by `project:intake` processing.

**Payload:**
```json
{
  "assignedSA": "daniel-kwon",
  "assignedPM": "string — PM slug",
  "discoveryQuestions": ["string — questions Daniel needs answered"],
  "technicalAssessment": "string — Daniel's initial read on complexity",
  "estimatedComplexity": "string — low | medium | high | very-high",
  "architectureNotes": "string — preliminary architecture direction"
}
```

**Process Rule:**
1. Update project phase to `scoping` in `.jware/projects/{projectId}/project.json`.
2. Dispatch Daniel Kwon agent to produce a detailed scope document:
   - Technical approach and architecture recommendation.
   - Effort estimate (in developer-weeks) with confidence level.
   - Risk register (assumptions that could change the estimate).
   - Recommended team composition.
3. If `estimatedComplexity` is `high` or `very-high`:
   - Dispatch Raj Patel (CTO) for architecture review of Daniel's preliminary notes.
   - Generate event: `security:review_requested` for Frank Morrison.
4. Dispatch the assigned PM to prepare the client-facing scope summary.
5. Create issuetracker issue: "Scoping: {projectTitle}" assigned to Daniel Kwon.
6. Write scope artifacts to `.jware/projects/{projectId}/scoping/`.
7. If Daniel's assessment surfaces questions requiring customer input:
   - Generate event: `decision:needed` with the discovery questions as payload.
8. Otherwise, generate event: `project:kickoff`.

**Generated Events:** `project:kickoff` (if no blockers), `decision:needed` (if customer input required), `security:review_requested` (if high complexity)

**Visibility Output:**

| Level 1 | Level 2 | Level 3 |
|---------|---------|---------|
| (nothing) | "Daniel Kwon is scoping '{title}' -- estimated {estimatedComplexity} complexity. {N} discovery questions pending." | Full scope document including Daniel's technical assessment, architecture notes, effort breakdown by component, risk register, and Raj's architecture review comments if applicable. |

---

#### `project:kickoff`

Development is beginning. The team has been assembled and the scope is locked.

**Trigger:** Scoping complete and all decisions received. Generated by `project:scoping` or `decision:received` processing.

**Payload:**
```json
{
  "scope": "string — finalized scope summary",
  "assignedDevLead": "string — dev lead slug",
  "teamMembers": ["string — developer slugs"],
  "sprintCount": "integer — estimated number of sprints",
  "milestones": [
    {
      "name": "string",
      "targetSprint": "integer",
      "deliverables": ["string"]
    }
  ],
  "architectureDecisions": ["string — key ADRs from scoping"]
}
```

**Process Rule:**
1. Update project phase to `active` in project.json.
2. Assign dev lead based on team availability and domain fit:
   - Backend-heavy or data pipeline work: Marcus Chen.
   - Frontend-heavy, UX-critical, or design-system work: Sarah Kim.
   - Infrastructure-heavy, performance-critical, or ops-heavy work: Tomas Rivera.
3. Dispatch the assigned dev lead to:
   - Break the scope into issuetracker issues (epics and tasks).
   - Assign initial tasks to team members.
   - Write the sprint 1 plan.
4. Dispatch Ben Hartley to finalize the project timeline with milestones.
5. Dispatch the assigned PM to send the kickoff summary to the customer.
6. Generate events: `task:created` for each issue the dev lead creates.
7. Generate event: `task:assigned` for each initial assignment.
8. Write kickoff record to `.jware/projects/{projectId}/kickoff.json`.

**Generated Events:** Multiple `task:created`, multiple `task:assigned`

**Visibility Output:**

| Level 1 | Level 2 | Level 3 |
|---------|---------|---------|
| "Project '{title}' is now in development." | "{assignedDevLead} is leading development on '{title}' with {N} team members. Sprint 1 begins with {M} tasks." | Full kickoff details: team roster with roles, sprint 1 task breakdown, milestone schedule, architecture decisions, and the dev lead's sprint plan. |

#### `project:milestone`

A significant progress point has been reached.

**Trigger:** Generated when a milestone's deliverables are all complete (tracked via `task:completed` events).

**Payload:**
```json
{
  "milestoneName": "string",
  "milestoneIndex": "integer — which milestone (1-indexed)",
  "totalMilestones": "integer",
  "completedDeliverables": ["string"],
  "sprintNumber": "integer — current sprint",
  "metricsSnapshot": {
    "tasksCompleted": "integer",
    "tasksRemaining": "integer",
    "defectsFound": "integer",
    "defectsResolved": "integer",
    "reviewPassRate": "float — percentage"
  }
}
```

**Process Rule:**
1. Write milestone record to `.jware/projects/{projectId}/milestones/{milestoneIndex}.json`.
2. Dispatch the assigned PM to prepare a milestone report for the customer.
3. Dispatch Ben Hartley to update the project timeline and assess schedule health.
4. If this is the final milestone:
   - Generate event: `project:review`.
5. If schedule variance exceeds 20% (behind):
   - Dispatch Diana Okafor (COO) for a delivery health assessment.
   - Dispatch the dev lead to propose a recovery plan.
6. Update `.jware/projects/{projectId}/project.json` with milestone completion data.

**Generated Events:** `project:review` (if final milestone)

**Visibility Output:**

| Level 1 | Level 2 | Level 3 |
|---------|---------|---------|
| "Milestone {milestoneIndex}/{totalMilestones} reached on '{title}'." | "Milestone '{milestoneName}' complete ({milestoneIndex}/{totalMilestones}). {tasksCompleted} tasks done, {tasksRemaining} remaining. Review pass rate: {reviewPassRate}%." | Full milestone report including deliverables list, metrics snapshot, schedule health assessment from Ben, PM's client-facing summary, and any COO commentary if schedule variance triggered review. |

---

#### `project:review`

The final review phase. All development is complete; the project is being evaluated for delivery.

**Trigger:** Generated by the final `project:milestone`.

**Payload:**
```json
{
  "reviewers": ["string — reviewer personality slugs"],
  "deliverables": ["string — final deliverable list"],
  "testSummary": {
    "totalTests": "integer",
    "passing": "integer",
    "failing": "integer",
    "coverage": "float — percentage"
  },
  "openDefects": "integer — count of unresolved defects",
  "securityStatus": "string — cleared | pending | issues-open"
}
```

**Process Rule:**
1. Update project phase to `review` in project.json.
2. Dispatch Raj Patel for final architecture review.
3. Dispatch Margaret Chen for final QA sign-off assessment.
4. Dispatch Frank Morrison for final security clearance.
5. If `openDefects > 0`:
   - Generate event: `qa:requested` for each open defect that has been marked fixed but not retested.
6. If `securityStatus` is `pending` or `issues-open`:
   - Generate event: `security:review_requested`.
7. Collect all reviewer verdicts. When all clear:
   - Generate event: `project:completed`.
8. If any reviewer rejects:
   - Generate the appropriate rejection events (`review:rejected`, `qa:failed`, `security:issue_found`).
   - Project remains in `review` phase until all blockers are resolved.
9. Write review record to `.jware/projects/{projectId}/review/`.

**Generated Events:** `project:completed` (if all clear), `qa:requested` / `security:review_requested` / `review:rejected` / `qa:failed` / `security:issue_found` (if issues found)

**Visibility Output:**

| Level 1 | Level 2 | Level 3 |
|---------|---------|---------|
| (nothing until verdict) | "'{title}' is in final review. Raj Patel reviewing architecture, Margaret Chen on QA, Frank Morrison on security." | Full review details: each reviewer's assessment, test summary, defect list, security findings, and the CTO's architecture verdict. |

---

#### `project:completed`

All work is done. The project has passed final review and is ready for delivery.

**Trigger:** All reviewers in `project:review` have cleared the project.

**Payload:**
```json
{
  "finalMetrics": {
    "totalSprints": "integer",
    "totalTasks": "integer",
    "totalDefectsFound": "integer",
    "totalDefectsResolved": "integer",
    "finalTestCoverage": "float",
    "reviewRejectionCount": "integer",
    "scheduleVariance": "float — percentage over/under estimate"
  },
  "deliveryNotes": "string — any delivery-specific notes",
  "teamRecognition": ["string — notable contributions"]
}
```

**Process Rule:**
1. Update project phase to `completed` in project.json with completion timestamp.
2. Dispatch the assigned PM to prepare the final delivery summary for the customer.
3. Dispatch Ben Hartley to write the project retrospective document.
4. Dispatch Diana Okafor to update capacity planning (team members are now available).
5. Archive project events to `.jware/projects/{projectId}/events-archive.json`.
6. Close all issuetracker issues for the project.
7. Write final metrics to `.jware/projects/{projectId}/final-metrics.json`.
8. If `scheduleVariance > 25%`:
   - Dispatch Daniel Kwon to review estimation accuracy for lessons learned.

**Generated Events:** None (terminal event).

**Visibility Output:**

| Level 1 | Level 2 | Level 3 |
|---------|---------|---------|
| "Project '{title}' is complete." | "'{title}' delivered. {totalSprints} sprints, {totalTasks} tasks, {finalTestCoverage}% test coverage. Schedule variance: {scheduleVariance}%." | Full project retrospective including final metrics, team recognition, estimation accuracy analysis, lessons learned, and the PM's delivery summary. |

---

#### `project:paused`

Work has been paused, either by the customer or due to a blocker.

**Trigger:** Customer requests a pause, or a `blocker:raised` event has been unresolved for more than 2 processing cycles.

**Payload:**
```json
{
  "reason": "string — why the project is paused",
  "pausedBy": "string — customer | system | personality slug",
  "blockerEventId": "string or null — the blocker event that caused the pause, if applicable",
  "resumeCondition": "string — what needs to happen for work to resume",
  "activeTasksAtPause": ["string — task IDs that were in progress"]
}
```

**Process Rule:**
1. Update project phase to `paused` in project.json.
2. Suspend all in-progress tasks: update their status to `paused` in issuetracker.
3. Dispatch Ben Hartley to recalculate project timeline with the pause factored in.
4. Dispatch the assigned PM to notify the customer (if pause was internal) or acknowledge the pause (if customer-initiated).
5. Write pause record to `.jware/projects/{projectId}/pauses/`.
6. If `pausedBy` is `customer`:
   - Dispatch Diana Okafor to assess resource reallocation during the pause.
7. No new task events are generated for this project until a `project:kickoff` or explicit resume event restores it to `active`.

**Generated Events:** None (events stop generating for paused projects until resumed).

**Visibility Output:**

| Level 1 | Level 2 | Level 3 |
|---------|---------|---------|
| "Project '{title}' is paused." | "'{title}' paused: {reason}. {N} tasks suspended. Resume condition: {resumeCondition}." | Full pause details including the blocker chain (if applicable), list of suspended tasks, Ben's revised timeline, Diana's resource reallocation notes, and the PM's customer communication. |

### 3.2 Task Lifecycle Events

---

#### `task:created`

A new task has been created in issuetracker.

**Trigger:** Dev lead breaks scope into tasks during kickoff, or a new task is identified mid-project.

**Payload:**
```json
{
  "taskId": "string — issuetracker issue ID",
  "title": "string — task title",
  "description": "string — task description",
  "createdBy": "string — personality slug who created it",
  "type": "string — feature | bug | chore | spike",
  "estimatedEffort": "string — hours or story points",
  "labels": ["string — relevant labels"],
  "parentEpic": "string or null — parent epic ID"
}
```

**Process Rule:**
1. Create the issue in issuetracker with the specified metadata.
2. Write task record to `.jware/projects/{projectId}/tasks/{taskId}.json`.
3. If the task was created by someone other than the assigned dev lead:
   - Notify the dev lead (log to project activity feed).
4. No agent dispatch -- task creation is a bookkeeping event.
5. Does not automatically generate `task:assigned`; the dev lead assigns tasks deliberately.

**Generated Events:** None (assignment is a separate event).

**Visibility Output:**

| Level 1 | Level 2 | Level 3 |
|---------|---------|---------|
| (nothing) | (nothing) | "Task created: '{title}' ({taskId}) by {createdBy}. Type: {type}, estimate: {estimatedEffort}." |

---

#### `task:assigned`

A developer has been assigned to a task.

**Trigger:** Dev lead assigns a task, or a task is reassigned after review rejection.

**Payload:**
```json
{
  "taskId": "string",
  "assignee": "string — developer personality slug",
  "assignedBy": "string — dev lead slug",
  "previousAssignee": "string or null — if reassignment",
  "reason": "string or null — reason for assignment/reassignment"
}
```

**Process Rule:**
1. Update the issuetracker issue: set assignee, status to `assigned`.
2. Update task record in `.jware/projects/{projectId}/tasks/{taskId}.json`.
3. If `previousAssignee` is not null (reassignment):
   - Add issuetracker comment from the dev lead explaining the reassignment (in their personality voice).
4. Check developer workload: if the assignee has 4+ in-progress tasks, log a capacity warning to the project activity feed.
5. No agent dispatch -- the developer picks up the task when `task:started` fires.

**Generated Events:** None directly. The developer starting work generates `task:started`.

**Visibility Output:**

| Level 1 | Level 2 | Level 3 |
|---------|---------|---------|
| (nothing) | (nothing) | "{assignedBy} assigned '{taskTitle}' to {assignee}." |

---

#### `task:started`

A developer has begun work on a task.

**Trigger:** Developer picks up an assigned task.

**Payload:**
```json
{
  "taskId": "string",
  "developer": "string — developer personality slug",
  "startedAt": "string — ISO 8601",
  "approach": "string — brief description of the developer's planned approach"
}
```

**Process Rule:**
1. Update issuetracker issue status to `in_progress`.
2. Dispatch the developer agent to begin implementation:
   - The agent works in the developer's personality voice and technical style.
   - The agent writes code, creates files, runs tests as appropriate.
   - The agent produces a PR description summarizing what was done.
3. Update task record with start timestamp and approach notes.
4. No generated events -- the developer agent works until the task is done, then generates `task:completed`.

**Generated Events:** `task:completed` (when the developer finishes).

**Visibility Output:**

| Level 1 | Level 2 | Level 3 |
|---------|---------|---------|
| (nothing) | (nothing) | "{developer} started work on '{taskTitle}'. Approach: {approach}." |

---

#### `task:completed`

A developer has finished implementation on a task.

**Trigger:** Developer agent completes implementation.

**Payload:**
```json
{
  "taskId": "string",
  "developer": "string — developer personality slug",
  "completedAt": "string — ISO 8601",
  "filesChanged": ["string — file paths"],
  "testResults": {
    "testsWritten": "integer",
    "testsPassing": "integer",
    "coverage": "float or null"
  },
  "prDescription": "string — PR summary",
  "selfAssessment": "string — developer's own assessment of the work"
}
```

**Process Rule:**
1. Update issuetracker issue status to `review_pending`.
2. Add issuetracker comment from the developer summarizing the implementation (in their personality voice).
3. Determine the code reviewer:
   - The dev lead of the developer's team reviews by default.
   - If the dev lead authored the task, another dev lead reviews (rotation: Marcus -> Sarah -> Tomas -> Marcus).
   - For security-sensitive tasks (tagged `security`): Frank Morrison also reviews.
4. Generate event: `review:requested` targeting the reviewer.
5. Update task record with completion data.
6. Check if this completion satisfies any milestone deliverables. If a milestone is now fully complete:
   - Generate event: `project:milestone`.

**Generated Events:** `review:requested`, possibly `project:milestone`

**Visibility Output:**

| Level 1 | Level 2 | Level 3 |
|---------|---------|---------|
| (nothing) | "{developer} completed '{taskTitle}'. PR ready for review." | Full PR description, files changed, test results, developer's self-assessment in their personality voice. |

#### `task:reassigned`

A task is being moved to a different developer.

**Trigger:** Dev lead decides to reassign (due to capacity, expertise, or review feedback suggesting a different developer).

**Payload:**
```json
{
  "taskId": "string",
  "fromDeveloper": "string — current assignee slug",
  "toDeveloper": "string — new assignee slug",
  "reassignedBy": "string — dev lead slug",
  "reason": "string — why the reassignment is happening"
}
```

**Process Rule:**
1. Update issuetracker issue: change assignee, add comment from the dev lead explaining the reassignment (in their personality voice).
2. Update task record.
3. If the original developer had work in progress:
   - Add issuetracker comment from the original developer summarizing what they completed and what remains (in their personality voice).
4. Generate event: `task:assigned` with the new developer.
5. If the reassignment is due to capacity:
   - Log capacity event to `.jware/projects/{projectId}/capacity-log.json`.

**Generated Events:** `task:assigned`

**Visibility Output:**

| Level 1 | Level 2 | Level 3 |
|---------|---------|---------|
| (nothing) | "'{taskTitle}' reassigned from {fromDeveloper} to {toDeveloper}. Reason: {reason}." | Full reassignment details including the dev lead's reasoning, the original developer's handoff notes, and any capacity implications. |

#### 3.2.5 Verification Events

---

#### `verify:requested`

JARVIS has been asked to verify a developer's completed work.

**Trigger:** Generated by `task:completed` processing (replaces direct `review:requested` generation from task:completed).

**Payload:**
```json
{
  "taskId": "string — the task being verified",
  "developer": "string — personality slug of the developer",
  "filesChanged": ["string — file paths changed by the developer"],
  "testResults": "object — test results from the developer agent's output",
  "branch": "string — git branch the work is on",
  "coverageThreshold": "number — minimum coverage percentage required",
  "attemptNumber": "number — which verification attempt this is (starts at 1)"
}
```

**Process Rule:**
1. Dispatch `jware-verifier` agent to run automated checks.
2. JARVIS runs: dependencies -> build -> lint/typecheck -> tests -> coverage (fail-fast order).
3. Post JARVIS result as issuetracker comment on the task issue.
4. If verdict PASSED: generate `verify:passed`.
5. If verdict FAILED: generate `verify:failed`.

**Generated Events:** `verify:passed` or `verify:failed`

**Visibility Output:**

| Level 1 | Level 2 | Level 3 |
|---------|---------|---------|
| (nothing) | "JARVIS verifying {developer}'s work on '{taskTitle}'." | Full JARVIS check output with voice line. |

---

#### `verify:passed`

JARVIS has cleared the code for human review.

**Trigger:** Generated by `verify:requested` processing when all checks pass.

**Payload:**
```json
{
  "taskId": "string",
  "testsPassed": "number — total tests that passed",
  "buildClean": "boolean",
  "lintClean": "boolean",
  "coveragePercent": "number — actual coverage achieved",
  "coverageThreshold": "number — required threshold",
  "duration": "string — how long checks took",
  "jarvisMessage": "string — the voice line summary"
}
```

**Process Rule:**
1. Update issuetracker issue status to `review_pending`.
2. Append "Cleared for review." to JARVIS issuetracker comment.
3. Determine code reviewer (dev lead default, rotate if dev lead authored, Frank Morrison for security-tagged).
4. Generate `review:requested` targeting the reviewer.

**Generated Events:** `review:requested`

**Visibility Output:**

| Level 1 | Level 2 | Level 3 |
|---------|---------|---------|
| (nothing) | "JARVIS: passed. Coverage {X}%. Moving to review." | Full JARVIS results with all check details and voice line. |

---

#### `verify:failed`

JARVIS found problems. Code returns to the developer or escalates.

**Trigger:** Generated by `verify:requested` processing when any check fails.

**Payload:**
```json
{
  "taskId": "string",
  "developer": "string — personality slug",
  "failures": [
    {
      "check": "string — dependencies|build|lint|tests|coverage",
      "details": "string — what failed and why",
      "severity": "string — error|warning"
    }
  ],
  "coveragePercent": "number|null — null if coverage check was skipped",
  "coverageThreshold": "number",
  "jarvisMessage": "string — the voice line",
  "attemptNumber": "number — which attempt this was"
}
```

**Process Rule:**
1. If `attemptNumber < 3`:
   - Update issuetracker issue status to `in_progress`.
   - Generate `task:assigned` back to original developer with JARVIS failure output in payload.
   - Increment `attemptNumber` for next cycle.
2. If `attemptNumber >= 3`:

## 5. Event Processing Loop

### Algorithm

```
function processEventQueue():
    queue = readFile(".jware/events.json")
    state = readFile(".jware/state.json")

    unprocessed = queue.filter(e => !e.processed)
    if unprocessed.length == 0:
        return "No events to process."

    # Sort by priority first, then by timestamp within the same priority
    sorted = sortEvents(unprocessed)

    newEventsGenerated = []
    cycleCount = 0
    MAX_CYCLES = 50  # absolute limit per /jware-auto invocation

    while sorted.length > 0 and cycleCount < MAX_CYCLES:
        event = sorted.shift()  # take the highest-priority, oldest event

        # Depth check
        if event.depth > MAX_DEPTH (10):
            markEventAsError(event, "Max recursion depth exceeded")
            continue

        # Look up the process rule for this event type
        rule = PROCESS_RULES[event.type]
        if rule is null:
            markEventAsError(event, "Unknown event type: " + event.type)
            continue

        # Execute the process rule
        result = executeRule(rule, event, state)

        # Mark the event as processed
        event.processed = true
        event.processedAt = now()
        event.processedBy = result.processedBy
        event.generatedEvents = result.newEventIds

        # Append generated events to the queue
        for newEvent in result.newEvents:
            newEvent.parentEventId = event.id
            newEvent.depth = event.depth + 1
            queue.push(newEvent)
            newEventsGenerated.push(newEvent)

        # Atomic write: update the queue file after each event
        atomicWrite(".jware/events.json", queue)

        # Re-sort to incorporate new events by priority
        sorted = queue.filter(e => !e.processed)
        sorted = sortEvents(sorted)

        cycleCount++

    # Update project state
    updateProjectPhases(state, queue)
    atomicWrite(".jware/state.json", state)

    return {
        processed: cycleCount,
        newEventsGenerated: newEventsGenerated.length,
        remaining: queue.filter(e => !e.processed).length
    }
```

### Sort Order

```
function sortEvents(events):
    PRIORITY_ORDER = {
        "CRITICAL": 0,
        "HIGH": 1,
        "MEDIUM": 2,
        "NORMAL": 3,
        "LOW": 4
    }

    return events.sort((a, b) => {
        # First sort by priority (lower number = higher priority)
        priorityDiff = PRIORITY_ORDER[a.priority] - PRIORITY_ORDER[b.priority]
        if priorityDiff != 0:
            return priorityDiff

        # Within the same priority, sort by timestamp (oldest first)
        return Date.parse(a.timestamp) - Date.parse(b.timestamp)
    })
```

### Atomic Write

```
function atomicWrite(filepath, data):
    tempPath = filepath + ".tmp." + randomId()
    writeFile(tempPath, JSON.stringify(data, null, 2))
    rename(tempPath, filepath)  # atomic on POSIX systems
```

### Cycle Limits

| Limit | Value | Rationale |
|-------|-------|-----------|
| Max depth per event chain | 10 | Prevents infinite recursive event generation. A `task:completed` -> `review:requested` -> `review:rejected` -> `task:assigned` -> `task:started` -> `task:completed` chain is 5 deep. 10 allows for a full rejection-fix-retest cycle. |
| Max events per `/jware-auto` | 50 | Prevents runaway processing. A typical project kickoff with 8 tasks generates ~24 events (8 created + 8 assigned + 8 started). 50 handles this with headroom. |
| Stale event threshold | 72 hours | Events unprocessed for more than 72 hours are flagged as stale in the queue. They are still processed but generate a warning. |

### Priority Escalation Rules

An event's priority can be escalated above its default:

1. **Repeated failures escalate.** A `qa:failed` event where `failureCount >= 3` escalates from MEDIUM to HIGH.
2. **Critical severity escalates.** Any event whose payload contains `severity: critical` escalates to CRITICAL.
3. **Cascade escalation.** If a HIGH event generates a child event, the child inherits at least MEDIUM priority (cannot be downgraded below the parent's level minus one).
4. **Stale events escalate.** Events unprocessed for 48+ hours escalate by one level (NORMAL -> MEDIUM, MEDIUM -> HIGH, etc.).

## 8. Error Handling and Recovery

### Agent Failure

**Scenario:** A dispatched personality agent fails mid-task (crash, timeout, context limit).

**Handling:**
1. The event that dispatched the agent remains marked as `processed` (the dispatch happened).
2. The engine generates a new event: the same type with `source: system` and a payload flag `retryOf: {originalEventId}`.
3. Retry limit: 2 retries per event. After 2 retries:
   - Generate event: `blocker:raised` with `blockerType: technical`.
   - Log the failure to `.jware/errors/{timestamp}-{eventId}.json`.
   - The task is moved to `blocked` status in issuetracker.
4. The dev lead is notified via an issuetracker comment.

### State Corruption

**Scenario:** `.jware/events.json` or `.jware/state.json` is corrupted or missing.

**Handling:**
1. **Missing events.json:** Create a new empty queue `[]`. Log the recovery event. All previous events are lost but the system can continue. The issuetracker remains the source of truth for task state.
2. **Corrupted events.json (invalid JSON):**
   - Check for `.jware/events.json.bak` (the previous atomic write backup).
   - If backup exists, restore from backup. Events processed between the backup and the corruption are replayed.
   - If no backup, create fresh queue. Scan issuetracker to reconstruct approximate state.
3. **Missing state.json:** Rebuild from events.json by scanning processed events to determine current project phases. Write reconstructed state.
4. **Corrupted state.json:** Rebuild from events.json as above.

### Event Pile-Up

**Scenario:** Events accumulate faster than they can be processed (e.g., a large project kickoff generates 30+ events in one cycle).

**Handling:**
1. The 50-event-per-cycle limit prevents unbounded processing.
2. If the queue exceeds 50 unprocessed events after a cycle completes:
   - Log a capacity warning to `.jware/warnings/`.
   - The next `/jware-auto` invocation processes the next batch.
   - Priority sorting ensures HIGH and CRITICAL events are handled first even when the queue is deep.
3. The user is informed: "Processed 50 events this cycle. {N} events remaining in queue. Run /jware-auto again to process more."

### Session Recovery

**Scenario:** The user's session crashes or is interrupted mid-processing.

**Handling:**
1. Atomic writes mean the queue file is always in a valid state. Either the old version or the new version exists; never a partial write.
2. Events that were being processed when the crash occurred:
   - If the event was not yet marked `processed: true` in the file, it will be reprocessed on the next cycle. This is safe because process rules are designed to be idempotent (creating an issue that already exists is a no-op; updating a status to its current value is a no-op).
3. Agent dispatches that were in-flight when the crash occurred:
   - The agent's work may be partially completed.
   - The reprocessed event may dispatch the agent again.
   - The agent should check existing state before making changes (e.g., don't create a duplicate issuetracker issue).
4. On session recovery, the engine runs a consistency check:
   - Scan for events where `processed: true` but `generatedEvents` is empty and the process rule normally generates events. Flag these for manual review.

### Recursion Prevention

**Scenario:** An event's process rule generates an event that eventually re-triggers the original event type, creating an infinite loop.

**Handling:**
1. **Depth limit (hard stop):** Events at depth 10 are not processed. They are marked with `error: "Max depth exceeded"` and logged.
2. **Cycle detection:** Before processing an event, check if an event of the same type, same taskId, and same source has been processed in the current cycle within the last 3 depth levels. If so, skip it and log a cycle warning.
3. **Known safe chains:** The following chains are expected and safe:
   - `task:completed` -> `review:requested` -> `review:rejected` -> `task:assigned` -> `task:started` -> `task:completed` (a review-fix cycle, depth 5)
   - `qa:failed` -> `task:assigned` -> `task:started` -> `task:completed` -> `review:requested` -> ... (a QA-fix cycle, can reach depth 8-9 in pathological cases)
4. **Circuit breaker:** If the same taskId appears in 8+ events within a single processing cycle, the engine pauses processing for that task and generates `blocker:raised`.

### Stale Event Detection

**Scenario:** Events that have been in the queue unprocessed for an extended period.

**Handling:**
1. Events older than 48 hours and unprocessed: escalate priority by one level.
2. Events older than 72 hours and unprocessed: flag as stale, add a warning marker to the event object.
3. Events older than 168 hours (7 days) and unprocessed: mark as `abandoned`, log to `.jware/warnings/`, and do not process. The user is notified that stale events exist.

## 9. Concurrency Safety

### Design Assumption

JWare Solutions operates within a single Claude Code session at a time. The primary concurrency concern is not multiple simultaneous users, but multiple rapid invocations of `/jware-auto` and the possibility of background agents reading/writing shared files.

### Atomic Writes

All state file modifications use the atomic write pattern:

```
1. Write to a temporary file (.tmp.{random})
2. Rename the temporary file to the target path
3. POSIX rename() is atomic -- the file is either the old version or the new version, never partial
```

Files that use atomic writes:
- `.jware/events.json`
- `.jware/state.json`
- `.jware/projects/{projectId}/project.json`
- All task records, review records, and other structured data files

### Event Queue Append Safety

New events are never appended directly to the queue file. Instead:

1. Read the full queue into memory.
2. Append new events to the in-memory array.
3. Write the entire array back atomically.

This ensures the queue is always a valid JSON array. The tradeoff is that the queue file is rewritten on every event processing step, which is acceptable for the expected volume (dozens to low hundreds of events per project).

### File Locking

Explicit file locking is **not used**. The reasons:

1. Claude Code runs in a single-threaded execution model within a session. Two processing loops cannot run simultaneously within the same session.
2. Atomic rename provides sufficient safety against partial writes.
3. If two separate sessions somehow both attempted to process the queue (which the system design prevents), the atomic write would ensure one wins cleanly -- the other's changes would be overwritten. This is acceptable because:
   - Events are idempotent by design.
   - The "lost" processing would be re-executed on the next cycle.

### Issuetracker Safety

The issuetracker uses file-based storage (`.jware/issues/`). Writes to issuetracker files follow the same atomic write pattern. Issue creation checks for existing issues with the same task ID before creating duplicates.

### State Consistency Invariants

The engine maintains these invariants at the end of every processing cycle:

1. **Queue integrity:** `.jware/events.json` is valid JSON containing an array of event objects.
2. **Processed consistency:** Every event with `processed: true` has a non-null `processedAt` timestamp.
3. **Depth consistency:** Every event's `depth` equals its parent's `depth + 1`, or `0` if it has no parent.
4. **Phase consistency:** The project phase in `project.json` matches the most recent phase-changing event that has been processed.
5. **Generated event traceability:** Every event ID listed in any event's `generatedEvents` array exists in the queue.

If any invariant is violated, the engine logs the violation to `.jware/warnings/` and attempts self-repair by re-deriving the violated field from its source of truth.

## Appendix B: Example Event Chain

A complete chain showing a task moving through the full lifecycle:

```
1. project:kickoff
   -> generates task:created (depth 1)
   -> generates task:assigned to liam-kowalski (depth 1)

2. task:assigned
   (no new events -- Liam picks it up)

3. task:started (source: liam-kowalski, depth 2)
   -> Liam agent works on the implementation
   -> generates task:completed (depth 3)

4. task:completed (source: liam-kowalski)
   -> generates review:requested targeting marcus-chen (depth 4)

5. review:requested
   -> Marcus reviews the code
   -> Marcus rejects: test coverage at 62%, needs 80%
   -> generates review:rejected (depth 5)

6. review:rejected (source: marcus-chen)
   -> generates task:assigned back to liam-kowalski (depth 6)

7. task:assigned (reassignment with review context)
   -> Liam picks it up again

8. task:started (source: liam-kowalski, depth 7)
   -> Liam fixes the test coverage
   -> generates review:revised (depth 8)

9. review:revised (source: liam-kowalski)
   -> generates review:requested targeting marcus-chen (depth 9)

10. review:requested
    -> Marcus re-reviews
    -> Marcus approves
    -> generates review:passed (depth 10)

11. review:passed (source: marcus-chen)
    -> generates qa:requested (depth 10 -- at the limit)

12. qa:requested
    -> Margaret Chen tests
    -> generates qa:passed (this would be depth 11 -- BLOCKED by depth limit)
    -> Instead: remaining in queue, processed on next /jware-auto at depth 0 (reset)
```

**Depth Reset Rule:** When the processing cycle ends and new events remain in the queue from the previous cycle, their depth is reset to `0` for the next cycle. Depth is a per-cycle recursion guard, not a permanent property. The `parentEventId` chain preserves full lineage regardless of depth resets.
