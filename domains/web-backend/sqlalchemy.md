# Domain: SQLAlchemy
**Loaded when:** Agent is implementing or modifying database models, queries, session management, or migrations.
**Key concern:** `session.add()` is synchronous even in async SQLAlchemy. Do not use `AsyncMock` for it -- use `MagicMock`.

---

## SQLAlchemy 2.0 Async Patterns

### Session Lifecycle

Create a session per request, commit or rollback, then close. Never share sessions across requests. Use `async_sessionmaker(engine, class_=AsyncSession, expire_on_commit=False)` and yield sessions via a FastAPI dependency.

### Sync vs Async Methods

This is the most common source of mock errors. Some session methods are synchronous even in async context:

| Method | Sync or Async | Mock with |
|---|---|---|
| `session.add(obj)` | **Sync** | `MagicMock` |
| `session.add_all(objs)` | **Sync** | `MagicMock` |
| `session.delete(obj)` | **Sync** | `MagicMock` |
| `session.expire(obj)` | **Sync** | `MagicMock` |
| `session.get(Model, id)` | **Async** | `AsyncMock` |
| `session.execute(stmt)` | **Async** | `AsyncMock` |
| `session.commit()` | **Async** | `AsyncMock` |
| `session.rollback()` | **Async** | `AsyncMock` |
| `session.flush()` | **Async** | `AsyncMock` |
| `session.refresh(obj)` | **Async** | `AsyncMock` |
| `session.close()` | **Async** | `AsyncMock` |

```python
# WRONG: AsyncMock for a sync method -- causes "coroutine never awaited" warning
mock_session.add = AsyncMock()

# RIGHT
mock_session.add = MagicMock()
mock_session.commit = AsyncMock()
mock_session.execute = AsyncMock(return_value=mock_result)
```

### Relationship Loading

Lazy loading does not work in async context -- it raises `MissingGreenlet`. Always use eager loading:

| Strategy | When to use |
|---|---|
| `selectinload` | One-to-many (emits separate SELECT IN) |
| `joinedload` | Many-to-one or one-to-one (emits JOIN) |
| `subqueryload` | Large collections where IN clause would be too big |

### safe_persist Pattern

`session.add(obj)` (sync, no await) followed by `await session.commit()` in a try/except that calls `await session.rollback()` on `IntegrityError`.

## Migrations with Alembic

### Migrations (Alembic)

Always review autogenerated migrations. Alembic cannot detect renamed columns (sees drop + add), data migrations, or index changes on existing data. Include both `upgrade()` and `downgrade()`.

## Testing Patterns

**Integration tests:** Use real sessions against a test database. Rollback in fixture teardown.

**Unit tests:** Mock sessions with correct sync/async types per the table above. Use `MagicMock(spec=AsyncSession)` as the base.

## Common Pitfalls

| Pitfall | Why it breaks | Fix |
|---|---|---|
| `AsyncMock` for `session.add()` | "coroutine never awaited" or silent failure | Use `MagicMock` for sync methods |
| Lazy loading in async | `MissingGreenlet` exception at runtime | Use `selectinload`/`joinedload` |
| Shared session across requests | Data leaks, stale reads, commit conflicts | Create session per request via DI |
| Missing `expire_on_commit=False` | Objects expire after commit, causing lazy load errors | Set in session factory |
| Autogenerated migration without review | Column renames become drop + add (data loss) | Always review before applying |
| No rollback in test fixtures | Test data persists, pollutes other tests | Rollback in fixture teardown |
